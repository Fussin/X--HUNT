# Core Proxy Architecture

This document describes the high-level architecture of the Core Proxy Engine.

## Components

The proxy engine is composed of the following main components:

*   **Server (`internal/proxy/server.go`)**: The main proxy server that listens for incoming connections and manages the connection lifecycle. It uses a pool of goroutines to handle multiple connections concurrently.

*   **TLS Manager (`internal/tls/manager.go`)**: The TLS manager is responsible for performing TLS interception (MITM). It includes a Certificate Authority (CA) that generates certificates on the fly for the hosts that are being intercepted. It also caches generated certificates to improve performance.

*   **Protocol Parsers (TBD)**: These components will be responsible for parsing the specific protocols, starting with HTTP/1.1, and then expanding to HTTP/2, HTTP/3, WebSocket, and gRPC.

*   **Transform Engine (TBD)**: This engine will allow for modifying traffic on the fly using a set of rules or scripts.

*   **Session Store (TBD)**: This component will be responsible for storing traffic for later analysis and replay.

## Data Flow

1.  A client connects to the proxy server's listener.
2.  The server accepts the connection and hands it off to a new goroutine.
3.  The `handleConnection` function initiates a TLS handshake with the client using a certificate generated by the TLS manager.
4.  Once the TLS handshake is complete, the proxy can read the decrypted traffic from the client.
5.  (TBD) The traffic is parsed by the appropriate protocol parser.
6.  (TBD) The parsed traffic can be modified by the transform engine.
7.  (TBD) The proxy establishes a connection to the target server and sends the (potentially modified) request.
8.  (TBD) The response from the target server is read, potentially modified, and then sent back to the client.
